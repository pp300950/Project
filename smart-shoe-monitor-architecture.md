# สรุปสถาปัตยกรรมระบบ — Smart Shoe Monitor (โมเดลจิ๋วสำหรับนำเสนอ)

## ภาพรวม Scope ของโมเดลจิ๋ว

| องค์ประกอบ | จำนวนในโมเดลจิ๋ว | จำนวนจริงตอนใช้งานจริง |
|---|---|---|
| รองเท้า (ESP-01) | 1 คู่ (ซ้าย+ขวา) | เท่าจำนวนผู้ป่วย |
| ESP8266 มาสเตอร์ | 1 ตัว ต่อสาย ultrasonic 3 เส้นออกไป | 3 ตัว แยกติดตั้งคนละจุด (สระว่ายน้ำ/ประตู/รั้วบ้าน) เพราะลากสายไกลไม่ได้ |
| Server | 1 ชุด (FastAPI บน Render) | เหมือนกัน |

---

## ชั้นที่ 1: ESP-01 (ฝั่งรองเท้า x2 ตัว)

หน้าที่: sensor เท่านั้น ไม่ตัดสินใจเรื่องเน็ตหรือ logic ใดๆ

- อ่านปุ่มกดใต้รองเท้า 1 ตัว/ข้าง → state digital 0/1 (1 = เหยียบ/ใส่รองเท้าอยู่, 0 = ไม่เหยียบ = ถอดรองเท้าหรือยกเท้าขึ้นจากพื้น)
- ส่งสถานะแบบ event-based (ส่งทันทีที่ state เปลี่ยน) + heartbeat เป็นระยะ เพื่อให้ฝั่งมาสเตอร์รู้ว่าออนไลน์อยู่
- มี ID ประจำตัวของตัวเอง เช่น `shoe01_left`, `shoe01_right` — ในอนาคตถ้ามีหลายมาสเตอร์ จะ "เชื่อฟัง" เฉพาะมาสเตอร์หมายเลขที่ตั้งไว้เท่านั้น (ในโมเดลจิ๋วนี้มีมาสเตอร์เดียวอยู่แล้วเลยไม่ติดปัญหานี้)

**WiFi:** ใช้สถาปัตยกรรมแบบ A — ESP-01 ต่อเข้ากับ AP ที่ ESP8266 มาสเตอร์ปล่อยเอง (วง local ไม่มีเน็ตจริง) ไม่ใช่ WiFi บ้าน/มือถือ ดังนั้น ESP-01 ไม่ต้องรู้จัก ไม่ต้องพึ่งพา WiFi ภายนอกเลย

**ตั้งค่าผ่าน WiFi config portal:**
- ชื่ออุปกรณ์/ID
- WiFi credentials ของ AP ฝั่งมาสเตอร์ (ค่า default อิงจากโค้ดตัวอย่างเดิมที่มีอยู่)
- Token แจ้งเตือน (เผื่อสลับจาก LINE ไปแอปอื่นในอนาคต)
- เปิดให้ตั้งค่าได้ 5 นาทีหลังบูต แล้วปิดอัตโนมัติ ไม่สนใจว่ามีคนตั้งค่าค้างอยู่หรือไม่

## ชั้นที่ 2: ESP8266 มาสเตอร์ (โมเดลจิ๋ว = 1 ตัว)

หน้าที่หลัก 4 อย่างพร้อมกัน:

1. **AP โหมด** — ปล่อยสัญญาณ WiFi ของตัวเองตลอดเวลา ให้ ESP-01 ทั้งสองข้างต่อเข้ามา ส่งสถานะรองเท้า
2. **STA โหมด** — ต่อเข้า WiFi บ้าน/มือถือ เพื่อขึ้นเน็ตไปหา server (ทำงาน 2 โหมดพร้อมกันแบบ `WIFI_AP_STA`)
3. **อ่าน ultrasonic sensor 3 ตัว** ที่ต่อสายตรงเข้ามา (จำลอง 3 จุด: สระว่ายน้ำ/ประตู/รั้วบ้าน) — ถ้าระยะที่วัดได้เปลี่ยนไป = มีคนเดินผ่าน → trigger เสียงแจ้งเตือนทันที (ทำงาน local ไม่ต้องรอ server)
4. **Mini dashboard ในตัว** — เข้าผ่าน WiFi วงเดียวกัน (ไม่ผ่านเน็ต) แสดงข้อมูลพื้นฐานแบบออฟไลน์ได้ ในกรณีที่เน็ตหลุดหรืออยากดูข้อมูลแบบเร็วๆ ในพื้นที่

**ตั้งค่าผ่าน WiFi config portal** (5 นาทีแรกหลังบูต เหมือน ESP-01):
- ระดับ server: เก็บไว้ฝั่ง ESP8266 (URL/endpoint ของ server, อาจมี API key)
- **`cloud_mode: on/off`** — ถ้า off จะส่งข้อมูลไปยัง local server (มือถือ) แทนการขึ้นคลาวด์ (ดูรายละเอียดในส่วน Data Security)

ส่งข้อมูลขึ้น server: ทุก event ที่เกิดขึ้น (สถานะรองเท้าเปลี่ยน + ultrasonic trigger + heartbeat) ส่งขึ้นไปให้ FastAPI ประมวลผลและเก็บลง database

## ชั้นที่ 3: Server (FastAPI บน Render)

- **ภาษา/เฟรมเวิร์ก:** Python + FastAPI (.py)
- **Host:** Render (free tier)
- **Database:** Supabase (Postgres free tier) — เลือกเพราะข้อมูลของระบบนี้มีโครงสร้างชัดเจนเป็นความสัมพันธ์ (ผู้ป่วย ↔ อุปกรณ์ ↔ เหตุการณ์) เหมาะกับ relational DB มากกว่า NoSQL แถมมี dashboard UI ในตัวสำหรับดูข้อมูลดิบเวลา debug ได้ทันทีโดยไม่ต้องเขียนเครื่องมือเสริม
- **หน้าที่:** รับข้อมูลจาก ESP8266 → ประมวลผล (เช่น คำนวณว่าเข้าเงื่อนไข "เคสเสี่ยงสูง" หรือไม่ ตามที่ออกแบบไว้ในแดชบอร์ดก่อนหน้านี้) → เก็บลง database
- **เผื่ออนาคต:** ช่องสำหรับต่อ AI chatbot API (ยังไม่ทำในเฟสนี้ แต่ออกแบบโครงสร้างให้เผื่อไว้)
- แดชบอร์ดเว็บที่ทำไปแล้วก่อนหน้านี้ (ไฟล์ `smart-shoe-dashboard.html`) จะมาดึงข้อมูลจาก API ของ server ตัวนี้แทนข้อมูล mock

## ชั้นที่ 4: ความปลอดภัยข้อมูลผู้ป่วย (Data Security & Privacy)

### หลักการเข้ารหัส

- **ข้อมูล "สถานะทั่วไป"** (รองเท้าใส่/ถอด 0/1, ultrasonic trigger, heartbeat, timestamp, device ID) → **ไม่ต้องเข้ารหัส** เพราะตัวมันเองไม่บ่งชี้ตัวตนผู้ป่วยโดยตรง (ตราบใดที่ device ID ไม่ใช่ชื่อจริง)
- **ข้อมูลที่ระบุตัวตนผู้ป่วยได้** (เช่น ชื่อ-นามสกุล) → เข้ารหัสด้วย **AES-256-GCM** ก่อนเก็บลง Supabase เสมอ
- **ใช้ AES ไม่ใช่ Hash (SHA-256)** เพราะ hash เป็นการเข้ารหัสทางเดียว ถอดกลับเป็นข้อมูลเดิมไม่ได้ เหมาะกับ "ตรวจสอบว่าตรงกันไหม" เท่านั้น ไม่เหมาะกับกรณีที่ต้องเรียกข้อมูลจริงกลับมาแสดงผลให้หมอ/ผู้ดูแลดู
- ใช้ GCM mode (ไม่ใช่ ECB/CBC เฉยๆ) เพราะให้ทั้งการเข้ารหัสและตรวจสอบความถูกต้องของข้อมูล (integrity) ในตัวเดียว แต่ละข้อความต้องมี IV/nonce ไม่ซ้ำกัน

### ที่เก็บ Key

- เก็บเป็น **Environment Variable บน Render** (เช่น `ENCRYPTION_KEY`) ทำหน้าที่แทน KMS แบบง่ายๆ ของระบบจิ๋วนี้ — ยังไม่มี KMS/HSM แยกเป็น service ต่างหาก
- ห้าม hardcode key ในโค้ด และห้าม commit ขึ้น Git (`.env` ต้องอยู่ใน `.gitignore`)

### Flow การเข้าถึงข้อมูล (อิงรูปแบบเดียวกับระบบโรงพยาบาลจริง ไม่ใช่ zero-knowledge)

1. Caregiver/หมอ login ด้วย username/password ปกติ (เก็บ hashed password ใน Supabase)
2. FastAPI ตรวจสอบสิทธิ์ (RBAC) ว่า user คนนี้มีสิทธิ์ดูข้อมูลผู้ป่วยรายนี้หรือไม่
3. ถ้าผ่าน → FastAPI อ่าน key จาก env var แล้ว decrypt ข้อมูลชั่วคราวในหน่วยความจำ (in-use) ส่ง plaintext กลับผ่าน HTTPS ให้ caregiver
4. บันทึกลงตาราง **audit log** ทุกครั้งที่มีการเข้าถึงข้อมูล (ใคร ดูข้อมูลใคร เมื่อไหร่)

> เหตุผลที่เลือกแบบนี้แทน "ให้ caregiver ถือ key เอง/จำ key เอง": ในงานจริงมีผู้ใช้หลายคน (หมอเวร พยาบาล ญาติ) ต้องเข้าถึงข้อมูลเดียวกัน การให้แต่ละคนจำ key เองไม่ scale และถ้าลืม key ข้อมูลจะกู้คืนไม่ได้เลย

### Trust Model

- **เชื่อใจ Render** — เป็นที่ถือ key และรัน decrypt logic
- **ไม่เชื่อใจ Supabase แบบเต็มที่** — เก็บแค่ ciphertext เท่านั้น ถ้า database ถูกขโมย/หลุดทั้งไฟล์ ข้อมูลที่ได้ไปก็ยังอ่านไม่ออก เพราะ key อยู่คนละที่ (defense in depth — แยก "ที่เก็บข้อมูล" ออกจาก "ที่ถือกุญแจ")
- ครอบคลุมความปลอดภัยทั้ง 3 สถานะของข้อมูล:
  - **At rest** (นิ่งอยู่ใน DB) → เข้ารหัสด้วย AES-256-GCM ✅
  - **In transit** (กำลังเดินทางผ่านเครือข่าย) → HTTPS/TLS ตลอดเส้นทาง ESP8266 ↔ Render ↔ Browser ✅
  - **In use** (กำลังถูกประมวลผล/แสดงผล) → เป็น plaintext ชั่วคราวใน RAM ของ server เท่านั้น จำกัดด้วย authentication + audit log — เป็นมาตรฐานเดียวกับระบบ healthcare ทั่วไป เพราะไม่มีระบบไหนเลี่ยง "in use" ได้ 100% ถ้ายังต้องประมวลผล/แสดงผลข้อมูล

### กรณีผู้ป่วยไม่ยินยอมส่งข้อมูลขึ้นคลาวด์เลย (Local-only Mode)

- เพิ่ม flag `cloud_mode: on/off` ในหน้า config ของ ESP8266 (ใช้โครงสร้างช่อง "server: URL/endpoint" เดิมที่มีอยู่แล้ว แค่สลับปลายทาง)
- ถ้า off → ESP8266 (ผ่าน STA ที่ต่อกับมือถือที่แชร์ฮอตสปอตอยู่แล้ว) ส่งข้อมูลไปยัง **local server บนมือถือเครื่องนั้นเอง** แทนการส่งไปหา Render
  - ฝั่งมือถือ (Android): ลง **Termux** รัน FastAPI + SQLite เป็น local server ในตัว ไม่ต้องมีอินเทอร์เน็ตจริงด้วยซ้ำ แค่เปิด hotspot ก็พอ
  - ข้อมูลทั้งหมดอยู่ในมือถือเครื่องเดียว ไม่ออกสู่อินเทอร์เน็ตเลย
  - เว็บแดชบอร์ดเดิมใช้ตัวเดิมได้ แค่เปลี่ยน base URL ของ API ให้ชี้มาที่ local แทน
- ข้อจำกัดที่ต้องรู้ไว้: Android ต้องตั้งให้รันแบบ foreground service + wake lock กัน OS kill ตอนจอดับ, ใช้แบตเยอะขึ้น, iOS ไม่มี Termux เทียบเท่า (ถ้าผู้ดูแลใช้ iPhone อาจต้องใช้ Raspberry Pi Zero แทน)

### ขอบเขตที่ยังไม่ทำในเฟสนี้ (รู้ไว้เผื่อกรรมการถามลึก)

- ยังใช้ **key เดียวกันทั้งระบบ** (ยังไม่แยก key ต่อผู้ป่วยแต่ละคน) — ถ้า key หลุด กระทบทุกผู้ป่วยพร้อมกัน เป็นจุดที่ควรปรับถ้า scale ไปใช้งานจริง
- ยังไม่มี **KMS/HSM แยกเป็น service ต่างหาก** (ของจริงระดับโรงพยาบาลใหญ่ใช้ เช่น AWS KMS / HashiCorp Vault) ตอนนี้ใช้ env var บน Render แทนไปก่อน
- ยังไม่ใช้ **Confidential Computing** (เช่น Trusted Execution Environment, AWS Nitro Enclaves) ที่ป้องกันแม้แต่ admin ของ server เองก็เห็น plaintext ไม่ได้ — เหมาะกับงานจริงระดับ enterprise มากกว่าโมเดลจิ๋ว

### ตารางเพิ่มเติมใน Supabase Schema

- `patients` — ข้อมูลผู้ป่วย โดย field ที่ระบุตัวตนได้ (ชื่อ ฯลฯ) เข้ารหัสด้วย AES-256-GCM
- `users` — บัญชี caregiver/หมอ (username, hashed password, role สำหรับ RBAC)
- `audit_log` — บันทึกการเข้าถึงข้อมูล (user_id, patient_id, timestamp, action)

## จุดที่ตกลงกันไว้แน่นอนแล้ว

- WiFi topology: แบบ A (ESP8266 แยกสองวง)
- Database: Supabase (Postgres)
- Server: FastAPI บน Render
- Sensor reading: ปุ่มกด digital 0/1, ultrasonic วัดระยะเปลี่ยน = trigger
- Config: เปิดตั้งค่าได้ 5 นาทีหลังบูตทั้ง ESP-01 และ ESP8266 แล้วปิดอัตโนมัติ
- ESP-01 = sensor บริสุทธิ์ ไม่ตัดสินใจเรื่องเน็ต, ESP8266 = ตัวกลางจัดการทุกอย่าง
- **ข้อมูลที่ระบุตัวตนผู้ป่วยเข้ารหัสด้วย AES-256-GCM ก่อนเก็บลง Supabase เสมอ (ไม่ใช้ hash)**
- **Key เก็บเป็น environment variable บน Render เท่านั้น ไม่ฝังในอุปกรณ์ ไม่ commit ขึ้น Git**
- **Caregiver/หมอ login ด้วย username/password + RBAC แยกออกจาก encryption key อย่างชัดเจน (ไม่ต้องจำ key เอง)**
- **มีตาราง audit log บันทึกการเข้าถึงข้อมูลทุกครั้ง**
- **รองรับโหมด local-only ผ่าน `cloud_mode: off` + มือถือ/Termux สำหรับผู้ป่วยที่ไม่ยินยอมส่งข้อมูลขึ้นคลาวด์**

พร้อมเริ่มลงโค้ดได้เลยครับ ถ้าจะเริ่ม อยากให้ผมเริ่มจากส่วนไหนก่อน — ESP-01, ESP8266 มาสเตอร์, หรือฝั่ง FastAPI/database schema (users + patients + audit_log)?
